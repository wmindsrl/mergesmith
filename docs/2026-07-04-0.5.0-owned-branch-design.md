# 0.5.0 — mergesmith owns the branch (spec-in-branch dispatch)

**Data:** 2026-07-04 · **Stato:** design approvato + POC validato · **Rilascio:** 0.5.0

## Problema

Due dolori reali, entrambi visti in produzione (port ramwms):

1. **Stallo iniziale invisibile.** Cursor crea *lui* il branch (`cursor/<slug>-<id>`). Se l'agent
   finisce il run senza pushare né aprire la PR (`pr:null`, nessun branch), il loop non ha nulla da
   sorvegliare: lo stato è ambiguo tra "sta lavorando", "morto senza consegnare", "ha pushato ma la
   PR è fallita". `reportStuckRuns` lo rileva grezzo ma il recovery è **manuale** (successo oggi con
   uncoded-verify: ri-dispatch a mano; e con inventory stanotte).

2. **Attrito di dispatch.** La spec deve stare su `origin/<base>` *prima* del dispatch (Cursor la
   legge dal suo clone). Da `rambase-agents` (worktree su `main`, senza le spec di `feat/ramwms-scaffold`)
   servono workaround fragili: materializzare la spec al path relativo + `specExistsOnBase` che fa
   `gh api contents` e rompe coi path assoluti. Correggere una spec dopo il dispatch richiede un push
   su scaffold + followup.

## Idea (di Marco) — mergesmith possiede il branch, la spec ci vive dentro

`dispatch <spec>`:
1. mergesmith crea il branch **deterministico** `fm.branch` dalla base, **committandoci dentro la spec**.
2. Dispatcha Cursor su quel branch con `workOnCurrentBranch: true` + `autoCreatePR: true`.
3. Cursor legge la spec dal branch, implementa, **apre lui la PR** verso la base.

mergesmith conosce **branch + spec-SHA da t=0**.

### Perché risolve entrambi

- **Stall detection binaria**: `run FINISHED && branchHead == specSha` = no-op certo → auto-recover.
  Nessun `pr:null` ambiguo.
- **Attrito azzerato**: la spec viaggia nel branch. Niente push-su-base preliminare, niente
  materialize, niente `specExistsOnBase`. Correggere una spec = commit sul branch + followup "re-read".
- **Igiene**: la spec atterra su base **insieme** al codice al merge (tracciabilità spec↔impl).

### POC (validato 2026-07-04, PR #123 su erisesrl/rambase, poi chiusa)

Branch `poc/mergesmith-branch-test` pre-creato con spec dentro → dispatch
`workOnCurrentBranch:true`+`autoCreatePR:true`. Risultato: Cursor ha (a) committato **su quel branch**
(non un figlio), (b) aperto la PR **da lì** verso `feat/ramwms-scaffold`, (c) la spec compare nel diff.
`branchHead (cf508b8) ≠ specSha (8fbf8a9)` → segnale di consegna corretto. **L'unica incognita
bloccante (comportamento Cursor sulla combinazione) è risolta.**

## Implementazione

### `github.ts` — creazione branch via GitHub API (nessun checkout locale)

`createBranchWithSpec(config, branch, base, specRepoPath, specContent): string` (ritorna la commit SHA
= specSha). Tutto via `gh api` (funziona da qualsiasi cwd):
1. base ref → `repos/{repo}/git/ref/heads/{base}` → commit SHA + tree SHA.
2. blob spec → `POST git/blobs` (content utf-8).
3. tree → `POST git/trees` (`base_tree` = base tree, entry `{path: specRepoPath, mode:100644, type:blob, sha:blob}`).
4. commit → `POST git/commits` (`tree`, `parents:[baseSha]`, message `spec: <id>`).
5. ref → `POST git/refs` (`refs/heads/<branch>`), oppure `PATCH git/refs/heads/<branch>` `force:true` se esiste
   (collisione da re-dispatch → reset del ref alla nuova spec-commit).

`branchExists(config, branch)` helper. Rimuovere/deprecare `specExistsOnBase` (non più necessario).

### `providers/types.ts` + `cursor.ts` — dispatch su branch dato

`DispatchInput` acquista `branch?: string`. In `cursor.dispatch`, se `input.branch` è presente:
`repos:[{url, startingRef: input.branch}]`, `workOnCurrentBranch: true`, `autoCreatePR: true`.
Altrimenti comportamento legacy (`startingRef: base`, `workOnCurrentBranch` assente, `autoCreatePR:true`).
Il poll branch/PR resta invariato (li scopre uguale). `DispatchResult` non cambia (branch già noto lato caller).

### `orchestrator/dispatch.ts` — nuovo flusso

```
fm = parseSpecFrontmatter(readFileSync(specPath))
branch = fm.branch                        // deterministico dal frontmatter (obbligatorio in 0.5.0)
specRepoPath = fm.specPath ?? `docs/superpowers/specs/<basename(specPath)>`
specSha = createBranchWithSpec(config, branch, fm.base, specRepoPath, specContent)
result = impl.dispatch({ prompt, repo, base: fm.base, model, branch })  // prompt: "Read <specRepoPath> in this branch…"
state.runs[fm.id] = { …, branch, specSha, base: fm.base }
```

Il prompt indica il path della spec **nel branch** (`specRepoPath`). Niente più `specExistsOnBase`.

### `orchestrator/state.ts` — `specSha`

`RunRecord.specSha?: string`. `setRework` invariato. Nuovo helper: `runsAwaitingPr(repo)` = run non-done,
con `branch` noto, senza PR ancora osservata (per la stall detection sotto).

### `orchestrator/tick.ts` — stall detection iniziale + auto-recover

Nuova funzione `sweepDispatchStalls(config, impl)` chiamata a fine tick (accanto a `reportStuckRuns`):
per ogni run con `branch` + `specSha` e **senza PR aperta/mergiata** (nessun PR open sul branch,
non in `reviewed`):
- se `agentIdle(ref)` (run FINISHED/error/expired) **e** `branchHead(branch) == specSha` → **no-op stall**.
  Auto-recover: `impl.adoptBranch(repo, branch, rediPrompt)` (max N tentativi via un `recoverAttempts` in state),
  poi needs-human. `branchHead` via `gh api repos/{repo}/git/ref/heads/{branch}` (assente = branch sparito → needs-human).
- se `branchHead != specSha` ma niente PR → l'agent ha committato ma non ha aperto la PR: followup
  "apri la PR verso <base>, ready for review" (riusa il ref).

`adoptBranch` (0.4.0) già spawna un agent fresco su un branch esistente — perfetto per il re-dispatch
idempotente (stesso branch, nessun `cursor/…-xyz` nuovo).

### Fix indipendente incluso — `mergeable == UNKNOWN` non è needs-human

In `act.ts`, ramo APPROVE-merge-fallito: oggi solo `CONFLICTING` → rebase, tutto il resto (incluso
`UNKNOWN`, transitorio: GitHub sta ancora calcolando la mergeability) → needs-human. Fix: su `UNKNOWN`
**ritenta al tick dopo** (non marcare `reviewed`, non flaggare) invece di escalare. needs-human resta
solo per fallimenti merge genuini con `mergeable == MERGEABLE`. (Falso needs-human visto oggi su #119.)

## Compatibilità

- Branch `cursor/*` esistenti / PR già aperte: gestiti come prima (il tick matcha per prefix + known branches).
  `sweepDispatchStalls` agisce solo su run con `specSha` (i nuovi). Nessuna migrazione dello state.
- `fm.branch` diventa **obbligatorio** per il dispatch spec (già presente in tutte le spec ramwms).
- Il verifier vede la spec nel diff: aggiungere una riga al CONTRACT/prompt → "il file spec nel diff è
  contesto, non codice da rivedere".

## Fuori scope 0.5.0

- PR aperta da mergesmith (resta a Curso) — deciso: le ergonomie/summary di Cursor valgono, e evitiamo la PR vuota.
- Provider non-Cursor: `branch`/`workOnCurrentBranch` sono capability opzionali; un provider che non le
  espone ricade sul flusso legacy.
- Riscrittura di `reportStuckRuns` (resta per i run senza `specSha`).

## Rollout

0.5.0: github.ts (createBranchWithSpec + branchExists), types/cursor (branch param), dispatch.ts (nuovo flusso),
state.ts (specSha + runsAwaitingPr), tick.ts (sweepDispatchStalls), act.ts (UNKNOWN retry). Unit test per
createBranchWithSpec (mock gh) + sweepDispatchStalls (stall vs consegnato vs no-PR). Poi rilascio + bump cron rambase.
