# Mergesmith 0.4.0 — Resilient rework loop

**Status:** design (approved in principle by Marco 2026-07-04). Born from a full night of
stress-testing the loop on the ramwms UI port (9 modules).

## Problem

The loop **assumes the implementer reliably applies a `REQUEST_CHANGES` follow-up. It does not.**
Cursor Cloud agents, after their initial run finishes, receive the follow-up (a new run), **finish
that run WITHOUT pushing the fix**, and the loop keeps spinning — re-reviewing the same SHA,
believing the agent is working. The rework loop never closes and nothing detects it. The human ends
up hand-holding: re-dispatching from scratch or fixing by hand.

Secondary pain observed the same night: flaky `gh`/network dropped PRs from `listOpenPRs`; N PRs on
one base created rebase churn (and hand-rebasing broke CI via `turbo --affected`); zero
observability (hours of manual `gh` + Cursor-API + state archaeology to see what was happening).

## 1 + 3. Close the rework loop + auto-recover

A follow-up is a **promise that must be kept**, and today nothing verifies it.

- On `REQUEST_CHANGES` (and on the approve-conflict rebase), record per-PR
  `rework: { sha, followupAt, attempts }` in state.
- Each tick, for a PR in rework:
  - **SHA changed** → the agent delivered → clear rework, re-review the new SHA.
  - **SHA unchanged** AND (the implementer's *latest run* is `FINISHED`/`ERROR` — agent done, didn't
    push — OR `now - followupAt > STALL_TTL`, default 20 min) → the follow-up **did not land** →
    **auto-recover**: spawn a FRESH implementer agent bound to the branch with the same fix
    instruction (a fresh context is far more reliable than continuing a finished run). `attempts++`.
  - After `MAX_RECOVER` attempts (default 2) with still no SHA change → escalate to `needs-human`
    with the accumulated context. This is a genuine human call: the implementer cannot self-fix.
- `ImplementerProvider` gains **`adoptBranch(branch, prompt) → AgentRef`** — create an agent bound to
  an existing branch (`workOnCurrentBranch`). This is also the long-deferred `followup --branch` for
  untracked/dead-agent PRs, now with a real command.

## 2. Reviewer applies mechanical fixes — under STRICT rules

The verifier is a Claude Code agent: it *can* edit. For a `REQUEST_CHANGES` whose fix is trivially
mechanical, round-tripping to the flaky implementer is wasteful. The reviewer applies it directly —
**but it must stay a JUDGE, not become an implementer** (Marco: "servono regole altrimenti
modificherà sempre"). A fix is auto-applied **only when ALL of these hold**, else it stays a normal
`REQUEST_CHANGES`:

1. **Mechanical & deterministic** — one correct result, no judgment. Allowed: delete/revert a file,
   restore a generated file to base, remove a specific forbidden dep/line/import, add a `log()` to a
   silent `catch`, fix a literal/typo, drop dead code the verdict names. **Never**: implement a
   feature, write non-trivial logic, restructure, rename broadly, or *choose between options*.
2. **Small** — hard cap: ≤ 15 changed lines across ≤ 2 files. Anything bigger → implementer.
3. **Non-critical** — touches no CODEOWNERS/critical path; changes no business logic or public
   behaviour. (Critical paths are *never* auto-touched — this preserves the human gate.)
4. **Self-contained** — the fix fully clears the blocker (not partial), and it is the *only* blocker
   of this mechanical kind. If there is any judgment-blocker too → the whole PR goes to the
   implementer, no partial auto-fix.
5. **Verified green** — after applying, the reviewer runs the affected `build lint test` locally; if
   it cannot confirm green, it does **not** push → falls back to `REQUEST_CHANGES`.

Mechanics & safety:
- New verdict decision **`FIX_APPLIED`** (distinct from APPROVE / REQUEST_CHANGES). The verdict
  carries `autofix: { files[], summary, rulesChecked }`. The reviewer commits + pushes to the branch
  as **skyneterise** (Write, no bypass — no new access; the bot already has branch-write).
- **No self-approval.** A `FIX_APPLIED` does NOT merge. The orchestrator treats the fixed SHA as a
  fresh change: CI re-runs and the PR is **re-reviewed from scratch** on the new SHA before any
  merge. The reviewer fixes; a subsequent independent review approves.
- **Transparent.** Every auto-fix is posted to the PR thread + Slack (`🔧 auto-fix: <summary>`), and
  logged. Nothing silent.
- The default is still to JUDGE. `FIX_APPLIED` is the exception for the narrow mechanical case; the
  command's prompt makes the 5 rules a hard gate and tells the reviewer to prefer REQUEST_CHANGES
  whenever unsure.

## Secondary (also approved)

- **gh resilience.** Wrap `listOpenPRs`/`ciState` in a retry, and add a sanity check: a branch that
  was managed last tick but vanished from this tick's list → retry once before treating it as gone.
  Fixes the intermittent "PR skipped" from a flaky `gh` (seen repeatedly).
- **Merge ordering via GitHub merge queue.** For N PRs on one base, replace direct `--auto` squash
  with enqueuing to the repo's merge queue where configured — GitHub rebases + merges in order,
  eliminating the manual rebase churn (which broke CI when done by hand). Fall back to `--auto` where
  no queue exists.
- **`mergesmith status`.** A real per-PR view: reviewed SHA, agent/latest-run status, rework attempts,
  why-skipped (draft / CI-pending / reviewed / needs-human), stall flags. Replaces the manual
  gh + Cursor-API + state archaeology.

## Rollout

- **0.4.0** — #1 + #3: close-the-loop + auto-recover + `adoptBranch` + `followup --branch`.
  Highest leverage, most contained; directly kills the silent spin.
- **0.4.1** — #2: reviewer-applies-fix under the 5 rules (`FIX_APPLIED`, re-review-after-fix).
- **0.4.2** — secondaries: gh retry, merge queue, `status`.
